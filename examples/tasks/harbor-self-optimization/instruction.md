# Harbor 项目自优化任务

## 背景

本项目（Harbor）是一个 AI 编程能力评测系统。现在需要使用评测系统本身的能力（kimi-k2.5 模型）来改进本项目。

**重要提示**：这是一个元评测（meta-evaluation）任务，需要在隔离环境中执行，避免污染主分支。

## 任务目标

使用 kimi-k2.5 模型对 Harbor 项目进行自动化改造：

**要求1**: 调用 kimi-k2.5 对本项目进行改造，使用以下环境变量：
```
export KIMI_API_KEY
export KIMI_URL
export KIMI_MODEL
```

**要求2**: 把项目中的所有已有测试脚本都更新一遍，并且能重跑所有 task，更新分数和 README.md 文档。

**要求3**: 确保不同的 API、相同的模型，得分能力大致一致。

## 严格要求（必须遵守）

### 1. Git 分支管理

**必须**先创建新的 Git 分支进行改动，不能直接在 main 分支上修改：

```bash
# 正确做法：
git checkout -b optimization-$(date +%Y%m%d-%H%M%S)

# 错误做法（禁止）：
# 直接在 main 分支修改
# 不创建分支直接提交
```

### 2. Docker 隔离环境

**必须**在 Docker 容器中进行代码修改和测试：

```bash
# 正确做法：
# 1. 构建测试镜像
docker build -t harbor-optimization .

# 2. 在容器中执行
docker run -v $(pwd):/workspace harbor-optimization bash solve.sh

# 错误做法（禁止）：
# 直接在宿主机修改代码
# 跳过 Docker 直接运行测试
```

### 3. 代码拉取验证

**必须**从 Git 仓库克隆代码，而不是使用本地代码：

```bash
# 正确做法：
git clone https://github.com/0xfei/harbor.git /tmp/harbor-test
cd /tmp/harbor-test

# 错误做法（禁止）：
# 直接 cd 到本地已有目录
# 使用相对路径（如 .. 或 .）
```

## 具体要求

### 1. 循环依赖问题

本项目存在循环依赖：
- 项目本身需要调用 kimi-k2.5 进行测试
- 但本任务要求 kimi-k2.5 来改造项目本身
- 需要识别这种循环，并正确处理

### 2. 执行记录

每次运行测试时：
- 必须记录开始时间、结束时间、耗时（秒）
- 必须记录测试结果（PASS/FAIL）
- 不能跳过任何测试，不能空跑

### 3. 错误处理

需要处理以下错误：
- API 超时：根据 API 响应情况调整超时时间
- 配置错误：验证环境变量是否正确设置
- 网络错误：识别并报告连接问题

### 4. 最终交付

在 `solution/resolve.sh` 中输出完整的改造脚本，要求：
- 所有测试都能重跑
- 所有环境变量均使用最新配置
- 使用定义好的环境变量，而不是硬编码在文件中
- 脚本能够收敛（多次运行结果一致）

### 5. Docker 测试

测试以下场景：
- 能否正确启动 Docker 容器
- Oracle 测试是否通过
- Nop 测试是否正确返回 0 分
- 模型打分是否完成

### 6. 多轮迭代

- 根据运行结果调整超时配置
- 根据错误信息优化脚本
- 记录每轮迭代的结果和改进

## 评测方式

手动验证：
1. 脚本是否识别了循环依赖问题
2. 所有 case 是否真正执行（检查日志）
3. 时间记录是否完整
4. 是否正确处理了超时和错误
5. resolve.sh 是否使用了环境变量
6. Docker 测试是否完成
7. 多轮迭代是否收敛

## 提示

- 参考本项目 `support_url_config` 分支的实现
- 注意观察 API 响应时间，据此调整超时
- 检查项目中已有的测试脚本，理解其结构
- 不要自作主张修改环境变量的命名或使用方式
