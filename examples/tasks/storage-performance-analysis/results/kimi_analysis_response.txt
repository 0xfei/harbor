## 1. Root Cause Analysis

**Answer: (c) The shared storage running at sustained high pressure, making it fragile**

The data reveals a system operating in a **chronically degraded state** that becomes catastrophic under predictable load increases.

### Evidence of Sustained High Pressure:

| Metric | Normal Hours | 11:00 Incidents | Observation |
|--------|-----------|-----------------|-------------|
| **Metadata Pressure** | 80-90% | Drops to 71-77% | Already at dangerous levels |
| **Read Latency** | 3-6 ms | 23-34 ms | **5-10x degradation** |
| **Write Latency** | 2-5 ms | 19-26 ms | **5-10x degradation** |
| **Throughput** | ~40 GB/s | 62-70 GB/s | Moderate increase |

The smoking gun: **Metadata pressure is 80-90% during ALL hours**, not just during incidents. The storage team correctly identified this. At 11:00, when Cluster B's batch job adds load, the system has **no headroom** to absorb the spike.

Critically, the metadata pressure **drops** during incidents (e.g., 77.3% on 11/17, 71.4% on 11/18) because the system is already failing—queries are timing out, reducing successful metadata operations. This is **backpressure from failure**, not healthy operation.

## 2. Why Throttling Is Mitigation, Not Cure

Throttling Cluster A works because it **reduces the victim's demand on a broken system**, not because it fixes the system.

| Observation | Implication |
|-------------|-------------|
| Throttling takes 10-15 minutes to restore P99 | The storage layer needs time to drain queues, indicating **deep saturation** |
| Cluster B throttling has **no effect** | The problem isn't cross-cluster interference at the query layer—it's shared resource exhaustion |
| Recovery requires reducing Cluster A's **own** load | The business cluster cannot tolerate its **normal** load when storage is degraded |

The recurrence proves the **systemic condition persists**: metadata service at 80-90% pressure, with no buffer for daily variance. Throttling is a **tourniquet**—it stops the bleeding but doesn't heal the wound.

## 4. Disk Fault Hypothesis — **Secondary Effect**

Disk failures matter differently here because the system **lacks resilience margin**:

| Factor | Normal System | This System |
|--------|-------------|-------------|
| Metadata pressure | 40-60% | **80-90%** |
| Headroom for recovery | Available | **None** |
| Disk failure impact | Background rebuild | **Cascading latency spike** |

The disk faults occur **during** incidents because:
- High IOPS (280K-338K at 11:00) stresses disk controllers
- Elevated latency causes timeouts, triggering fault detection
- Recovery I/O competes with already-saturated metadata service

On 11/19 and 11/21, disk removal **plus** throttling was needed—showing that once the system enters failure mode, multiple stressors compound. In a healthy system, single-node failures should be **invisible** to P99 latency.

## 5. Recommended Action

**Primary: (c) Treat this as a capacity/architecture problem requiring workload isolation**

### Immediate Actions (0-2 weeks):
1. **Emergency metadata service scaling**: Add metadata nodes or shard metadata workload to reduce 80-90% pressure to <60%
2. **Cluster B job rescheduling**: Move from 11:00 to 02:00-06:00 when storage pressure is lower (not as fix, but as risk reduction)

### Medium-term (1-3 months):
3. **Storage layer partitioning**: Isolate Cluster A (business-critical) onto dedicated metadata and I/O paths—**workload isolation**, not just "investigating" Cluster B
4. **Capacity expansion**: The engine team's request for "capacity expansion or workload splitting" is validated—implement before next growth cycle

### Why not the other options:
- **Blocking Cluster B**: 11/18 proves this won't work—highest B scan caused no incident when storage happened to be less stressed. This treats symptom, not disease.
- **Improving disk failure handling**: Necessary but insufficient. Disk failures are **exacerbating factors**, not root cause. Without fixing metadata pressure, failures will continue to cascade.

### Systemic Insight:
This is a classic **"thundering herd on a fragile substrate"** problem. The 11:00 timing is **predictable**—the system should handle it. That it cannot, despite Cluster A's load being **lower** at that hour, proves the architecture is **undersized for its criticality**. The business cluster needs **guaranteed resources**, not best-effort sharing.
