## 2. Why Throttling Is Mitigation, Not Cure

Throttling Cluster A works because it **reduces the instantaneous metadata operation rate**, allowing the queue to drain. However:

- The problem recurs daily because **the underlying capacity constraint remains unchanged**
- Throttling addresses the symptom (query pile-up) not the disease (metadata service saturation)
- The system returns to the same 80–90% metadata pressure baseline, guaranteeing future incidents

This confirms the root cause is **structural capacity limitation**, not a tunable configuration issue or external trigger that can be eliminated.

## 4. Disk Fault Hypothesis — Why Failures Matter Differently Here

Disk faults occur during 4 of 7 incidents (11-17, 11-19, 11-21, 11-22), but notice:

- **11-19**: Fault at 11:08, incident started at 11:08 (simultaneous)
- **11-21**: Fault at 11:22, incident started at 10:58 (fault followed incident)
- **11-22**: Fault at 11:05, incident started at 11:03 (fault followed incident)

The temporal ordering shows disk faults are often **consequences or coincident effects**, not causes. The 11-19 simultaneous timing suggests the degraded storage state triggers both the incident and the fault detection.

**Why disk faults matter more in this architecture**:

In healthy distributed storage, single-node failures are handled gracefully because:
- Remaining nodes absorb the load with adequate headroom
- Metadata operations continue normally
- Rebuild traffic is throttled and non-disruptive

Here, the system lacks the **resilience margin** for graceful degradation. When a disk fails:
- Its load redistributes to already-saturated nodes
- Metadata pressure spikes further (rebuild requires heavy metadata operations)
- The queuing delay cascades immediately because there's no buffer capacity

The disk faults are **amplified by the saturated state**, not independent causes.

## 5. Recommended Action

**Correct long-term fix: Treat this as a capacity/architecture problem requiring workload isolation**

Specifically:

1. **Immediate**: Implement **dedicated storage isolation** for Cluster A (business-critical, user-facing) vs. Cluster B (batch analytics). The shared storage model creates unmanageable tail-risk dependencies.

2. **Short-term**: **Metadata service horizontal scaling** or sharding. The 80–90% sustained pressure indicates a bottleneck that cannot be tuned away.

3. **Medium-term**: **Cluster A capacity expansion** as the engine team requested. The data shows Cluster A's normal load (22–28 QPS, 5–7 GB/h scan) is already at the edge of what the shared infrastructure can sustain.

**Why not the other options?**

- **Blocking Cluster B's jobs**: The 11-18 anomaly proves this won't eliminate incidents. Cluster A can fail independently.
- **Improving disk failure handling**: This addresses a symptom amplifier, not the root cause. In a properly provisioned system, these faults would be non-events.

The fundamental issue is **architectural**: a user-facing latency-sensitive workload shares saturated infrastructure with batch analytics, creating systemic fragility that no operational mitigation can resolve.
